# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

PSEye — a free, community-first tracker for the Philippine Stock Exchange (PSE). pnpm + Turborepo monorepo.

## Commands

Run from the repo root unless noted.

- `pnpm install` — install all workspace deps
- `pnpm dev` — `turbo run dev` (starts `apps/web` on localhost:3000)
- `pnpm build` / `pnpm lint` / `pnpm typecheck` — run across all packages via Turbo (each depends on `^build` first)
- Scope any script to one package: `pnpm --filter @pseye/web <script>` (package names: `@pseye/web`, `@pseye/db`, `@pseye/etl`, `@pseye/source-quotes`, `@pseye/source-news`, `@pseye/source-corporate-actions`, `@pseye/source-block-sales`, `@pseye/source-disclosures`, `@pseye/source-offerings`, `@pseye/source-foreign-flow`, `@pseye/treemap-layout`, `@pseye/config`)
- `pnpm db:generate` — Drizzle Kit generate against `packages/db/src/schema.ts`; doesn't need `DATABASE_URL` (diffs against `packages/db/drizzle/`'s migration history). **2026-07: a real, populated `DATABASE_URL` now exists** (`apps/web/.env.local`) and migrations are applied through `0007_*.sql` via `pnpm --filter @pseye/etl migrate-db` (see its own doc comment — not `pnpm db:push`, whose interactive conflict-resolution prompt hangs in any non-interactive context, CI included; `migrate-db` just applies the already-generated, already-reviewed SQL files and is idempotent). After adding a migration, run `migrate-db` again to apply it — a new nullable column doesn't retroactively appear on rows already in the DB from before an ETL job's insert statement included it, so also re-run that job once locally (or switch its upsert from `onConflictDoNothing` to `onConflictDoUpdate` first, if the new column should backfill onto existing rows — see fetch-disclosures.ts/fetch-offerings.ts's `url` column backfill for the pattern).
- `pnpm --filter @pseye/etl fetch-quotes` / `fetch-news` / `fetch-corporate-actions` / `fetch-block-sales` / `fetch-disclosures` / `fetch-offerings` / `fetch-foreign-flow` / `fetch-historical-quotes` / `fetch-market-snapshot` / `backfill-company-profiles` / `migrate-db` — run an ETL job locally; all require `DATABASE_URL`. In production these run on a schedule via GitHub Actions, not on app request: `quotes-2x-daily.yml` and `market-snapshot-2x-daily.yml` run quotes/market-snapshot twice on weekdays (midday + after-close settle); `fetch-daily.yml` runs news/disclosures/corporate-actions/historical-quotes/offerings/block-sales all once/day in one job (shared checkout/install, `continue-on-error` per step so one bad source doesn't skip the rest); `foreign-flow-weekly.yml` runs weekly. **2026-07: these used to be 9 separate workflows, several hourly** (news/disclosures every hour, quotes/market-snapshot hourly during trading hours) — ~1,900 workflow runs/month, which blew past GitHub Actions' free monthly minutes on a private repo. Consolidated down to ~65 runs/month; don't reintroduce a per-source hourly schedule without checking the Actions usage page first. `backfill-company-profiles` and `migrate-db` are one-off/manual, no workflow schedule (see their own doc comments).
- `pnpm test` — `turbo run test`. Has real coverage now: `@pseye/source-quotes` (PSE Edge stock-data/chart-data HTML+JSON parsers, `PseEdgeQuoteSource` error handling), `@pseye/source-disclosures` and `@pseye/source-corporate-actions` (their PSE Edge HTML parsers), `@pseye/etl` (the foreign-flow PDF parser — see below), `@pseye/treemap-layout` (`computeLayout.ts`/`color.ts` pure math), and `@pseye/web` (`lib/dca.ts`'s DCA simulation/composite-index math). Scope with `pnpm --filter <pkg> test`.
- **CI**: `.github/workflows/ci.yml` runs `pnpm lint && pnpm typecheck && pnpm test && pnpm build` on every push to `master` and every PR. Every other workflow in `.github/workflows/` is schedule-only ETL jobs — this is the only one gating code changes.

## Architecture

Turborepo/pnpm workspaces: `apps/*`, `packages/*`, `packages/sources/*`, `etl`.

- **`apps/web`** — Next.js 16 (App Router) + React 19 frontend. Routes: `/` (`MarketMapPage`, the treemap "market map"), `/charts` (NASDAQ-only TradingView embed — see below), `/news`, `/calendar` (dividend & corporate actions), `/block-sales`, `/disclosures`, `/offerings` (IPO/follow-on tracker), `/foreign-flow`, and `/dca` (cost-averaging calculator). All are async Server Components using ISR (`export const revalidate`) rather than client fetching, except the DCA calculator's interactive core (`DcaCalculator`/`DcaChart`, client components) and the Charts page's TradingView widget (also client-side, third-party script). The DCA calculator now has a real API route — see `app/api/history/route.ts` below — it isn't purely in-browser anymore. `app/error.tsx`/`app/not-found.tsx` provide site-themed fallback UI (this Next version's `error.tsx` takes `unstable_retry`, not the classic `reset` — another breaking-change surprise, see below). `apps/web/AGENTS.md` / `CLAUDE.md` flags that this Next.js version has breaking changes vs. training data — check `node_modules/next/dist/docs/` before writing Next-specific code.
- **`/charts`** (`app/charts/page.tsx`, `components/TradingViewChart.tsx`) — embeds TradingView's free "Advanced Chart" widget. **TradingView's free embeddable widgets refuse to serve PSE-listed symbol data** — confirmed live across multiple widget types (Advanced Chart, Symbol Overview, Single Quote) and multiple PSE tickers plus the PSEi index; all return "This symbol is only available on TradingView" while the identical widget renders real data fine for e.g. `NASDAQ:AAPL`. Don't re-attempt PSE symbols in a TradingView widget expecting a config fix — it won't work. The ticker picker is scoped to a curated NASDAQ list instead; `allow_symbol_change` on the widget still lets a visitor search any other TradingView-supported symbol themselves. Also: TradingView's script overwrites `style.height="100%"` (inline, beats any Tailwind class) on the element carrying the `tradingview-widget-container` class — give a wrapper *outside* that class the fixed height, or the widget collapses to a tiny default size.
- **`apps/web/lib/*.ts`** — the "DB-backed when `DATABASE_URL` is set and populated, otherwise `Mock*Source`, falling back on any DB error too" pattern is now used by several files, not just quotes — keep this one-fallback-path-per-feature shape rather than instantiating a `Mock*Source`/real source directly in a page: `quotes.ts` (`getDailyQuotes`), `disclosures.ts` (`getDisclosures`), `corporateActions.ts` (`getCorporateActions`), `foreignFlow.ts` (`getForeignFlowPageData` — index-level and per-stock are both real now (2026-07), from two different weekly/daily ETL jobs, so they still fall back *independently* rather than sharing one flag; returns a `stockFlowSource: "real" | "mock"` tag so the page's "sample data" caveat only shows when the per-stock fallback actually fired), and `historicalQuotes.ts` (`getHistoricalQuotes` — falls back for *all* requested tickers together, not per-ticker, since blending a real close series with a synthetic one in the same composite average would be misleading; returns `{ source: "real" | "mock", history }` rather than bare history, so `DcaCalculator.tsx` can show its "sample data" caveat only when the fallback actually fired instead of unconditionally). `getDailyQuotes()`'s `Quote.price`/`pctChange` are `number | null` — null means PSE Edge had no trade to report (suspended ticker, no fill yet today); render that as "N/A", never coerce to 0.
- **`apps/web/app/api/history/route.ts`** — the app's first (and so far only) Route Handler. Backs the DCA calculator: `DcaCalculator.tsx` can't call a real `HistoricalQuoteSource` directly the way it called `MockHistoricalQuoteSource` before (no CORS to PSE Edge, and it would mean live-scraping per keystroke instead of the batch-job-writes-to-DB pattern every other feature uses) — so it `fetch()`es this route instead, which reads `historicalQuotes.ts` (our own DB, or mock fallback) server-side. If a future interactive client feature needs real per-request data, follow this same shape rather than reaching for a `*Source` from client code.
- **`packages/treemap-layout`** (`@pseye/treemap-layout`) — pure layout + color math, no React/DOM dependency (kept that way so it's reused by `app/opengraph-image.tsx`'s static Satori-rendered share image, per its own doc comments). `computeLayout.ts` groups stocks by sector and sizes boxes by market cap with `d3-hierarchy`'s squarified treemap; a stock whose market cap is tiny relative to its sector peers can otherwise `round(true)` down to an exactly-0px-wide/tall (invisible, unclickable) box, so every leaf rect's far edge is nudged out by 1px when it would round to under 1px — a fixed, tested invariant, not a cosmetic choice. `color.ts` builds the diverging (red↓ / green↑) `%`-change color scale with `d3-scale`/`d3-interpolate`, separately for light/dark mode, plus a WCAG-contrast helper (`getContrastText`) to pick label ink color. `ForeignFlowChart` reuses the same green/red finance convention for its own polarity metric (net buying/selling).
- **The market map's `?filter=` and `?ticker=` URL params** (`MarketMap.tsx`, `TreemapChart.tsx`) sync the active filter and the selected stock to the URL via `useSyncExternalStore` (a custom event + `popstate`, since `history.replaceState` doesn't dispatch either on its own) — both deliberately server-snapshot to a default (`"all"` / `null`) so hydration never mismatches a client landing on a deep-linked URL. This is what makes a specific stock or filter shareable via `ShareButton` (which just shares `window.location.href`). **Don't try to make `app/opengraph-image.tsx` vary per `?filter=`/`?ticker=`** — that file convention only receives dynamic route `params`, never `searchParams` (confirmed against `node_modules/next/dist/docs`), and wiring a matching share image in via `generateMetadata` reading `searchParams` on `page.tsx` flips the whole `/` route from `○ Static, revalidate 1h` to `ƒ Dynamic` (confirmed via a real `next build`) — i.e. every visit, not just shared links, would hit a live data fetch. Not worth that tradeoff for a cosmetic share-image improvement; the real fix — giving a per-company page its own `opengraph-image.tsx` — is exactly what `/stocks/[ticker]` (added 2026-07, see next bullet) now does, at no such cost.
- **`app/stocks/[ticker]/page.tsx`** — per-company page (price/%change/market-cap-rank/sector-rank stats, a real closing-price chart, company profile, recent disclosures, dividend/corporate-action history, and news mentions), added as the single biggest indexable-page-count lever available (9 flat routes → 97, one per `PSE_EDGE_COMPANIES` entry via `generateStaticParams`). Reuses every existing whole-list lib function (`getDailyQuotes`, `getCompanyProfiles`, `getDisclosures`, `getCorporateActions`, `getHistoricalQuotes`) filtered in-memory to the one ticker rather than adding new per-ticker DB queries — all of those lists are already small (≤97 rows, or a bounded lookback window) so this is cheap. Ticker lookup is case-insensitive against the roster, but the canonical `<link rel="canonical">` always points at the uppercase form, so `/stocks/bdo` and `/stocks/BDO` don't count as duplicate content. Has its own `opengraph-image.tsx` (gets `params.ticker` for free) and a `Corporation`/`tickerSymbol` JSON-LD block. `StockPriceChart` (a plain server component, no client JS) only ever renders when `getHistoricalQuotes` reports `source: "real"` for that ticker — never `MockHistoricalQuoteSource`'s synthetic walk, since a fake chart on a page framed as one company's real data would undermine the page's whole purpose. `apps/web/app/sitemap.ts` includes one entry per company for the same reason.
- **`TickerSearch`** (header, both desktop-inline and inside the mobile hamburger panel) — client-side typeahead over the bundled `PSE_EDGE_COMPANIES` roster (no per-keystroke network request; ~97 companies fits comfortably in memory) that navigates straight to `/stocks/[ticker]`. First real search UI anywhere in the app. `SiteHeader.tsx` collapses nav + search behind a hamburger below `sm` — nine nav links wrapping was an accepted tradeoff before, but adding a search box tipped it into needing a real mobile menu.
- **`packages/db`** (`@pseye/db`) — Drizzle ORM schema (`daily_quotes`, `news_items`, `corporate_actions`, `block_sales`, `disclosures`, `offerings`, `index_foreign_flow`, `stock_foreign_flow`, `historical_quotes`, `market_snapshot`, `company_profiles`) + Neon serverless Postgres client (`createDb`) + hand-written query helpers (`queries.ts`, e.g. `getLatestDailyQuotes`, `getHistoricalQuotes`, `getIndexForeignFlowHistory`). `drizzle.config.ts` reads `DATABASE_URL` — but `drizzle-kit generate` itself doesn't need a live DB (it diffs `schema.ts` against `packages/db/drizzle/`'s existing migration snapshots), so schema changes can be made and migrations generated even without `DATABASE_URL` set. See the `pnpm db:generate` bullet above for the current, real, populated-database state and how to apply new migrations. `daily_quotes.price`/`pct_change` are nullable — see `apps/web/lib/quotes.ts` above.
- **`packages/sources/*`** — one pluggable source interface per feature: `quotes` (`QuoteSource`, `HistoricalQuoteSource`), `news` (`NewsSource`), `corporate-actions` (`CorporateActionSource`), `block-sales` (`BlockSaleSource`), `disclosures` (`DisclosureSource`), `offerings` (`OfferingSource`), `foreign-flow` (`ForeignFlowSource`). Swap implementations behind the interface; don't change callers.
  - **Real, PSE Edge–backed** (same knowing legal tradeoff as each other — see docs/PLANNING.md's Open Question #1 — public pages/endpoints, not a licensed feed; each runs from a scheduled ETL job only, never called per-request from the web app): RSS news (`createRssSource`); **`quotes`'s `PseEdgeQuoteSource`** (`packages/sources/quotes/src/pseEdge/`) — per-company "Stock Data" pages (`stockData.do?cmpy_id=...`) for delayed/EOD price, % change, market cap; **`quotes`'s `PseEdgeHistoricalQuoteSource`** (same dir) — real daily OHLC via `/common/DisclosureCht.ax` (the JSON endpoint that page's own price chart calls; needs a `security_id` read off that same page's dropdown first, not just `cmpy_id`); **`disclosures`'s `PseEdgeDisclosureSource`** (`packages/sources/disclosures/src/pseEdge/`) — POSTs `/announcements/search.ax`, matches rows to our roster by `cmpy_id`; **`corporate-actions`'s `PseEdgeCorporateActionSource`** (`packages/sources/corporate-actions/src/pseEdge/`) — POSTs `dividends_and_rights_info_list.ax`, Dividends tab only (Rights tab is mostly "TBA" placeholder dates, not worth mapping yet). All of these key off `@pseye/source-quotes`'s static `PSE_EDGE_COMPANIES` ticker→`cmpy_id` roster (`pseEdgeCompanyDirectory.ts`), matching by `cmpy_id`/`security_id` rather than by company-name text (PSE Edge's own name strings occasionally differ in punctuation/casing).
  - **Real, but not PSE Edge — a PDF, not HTML**: index-level foreign flow, block sales, and (as of 2026-07) per-stock foreign flow too. See the `etl` bullet below — all three deliberately do *not* live in their `packages/sources/*` package (`foreign-flow`, `block-sales`).
  - **Still `Mock*Source` placeholders**, each investigated this project and found to have no free public source (don't re-investigate without new information): `offerings`'s `MockOfferingSource` — the one free real feed found (`disclosureData/listing_applicants_list.do`, "Listing Applicants") is currently empty ("no data.") and has no subscription-window dates at all, which the `offerings` DB table requires (`subscription_start`/`end` are `NOT NULL` and part of its unique key) — wiring it in for real would need a schema change to serve a source that's empty in production anyway. Re-investigated 2026-07 (disclosure search, SEC Philippines, PSE's own IPO/SRO pages) — still no viable source; SEC Philippines' site is fully bot-blocked (HTTP 403 on every endpoint, not just paywalled). (Per-stock foreign flow used to be here too, in the same "no free source" bucket as offerings — see the `etl/lib/pseQuotationReport/` bullet below for how a 2026-07 re-investigation turned up a real one. Re-investigating a "no free source" finding occasionally does turn up something the first pass missed — this project has now hit that twice, first for block sales, then for per-stock foreign flow.)
  - `MockHistoricalQuoteSource` takes the *current* quotes as a constructor param (not a fixed internal list) so its anchor price matches whatever `QuoteSource` the caller is using (DB or mock) — see `apps/web/lib/historicalQuotes.ts`. Its synthetic close is floored at ₱0.01 — a sub-centavo mock price (there's a ₱0.012 sample ticker) could otherwise round to an exact 0.00 and divide-by-zero in `simulateDca`/`buildCompositeHistory`.
- **`etl`** — standalone `tsx` scripts (no build step) invoked by the GitHub Actions workflows in `.github/workflows/` (one per job; cadence matches each real-world source's publish schedule — hourly during PSE trading hours for quotes, hourly for news/disclosures/market-snapshot, daily for corporate-actions/offerings/historical-quotes/block-sales, weekly for foreign-flow); each upserts source data into Postgres via `@pseye/db`, deduping on a natural key per table (ticker+date, URL, reference number, etc. — see each job for its `onConflictDoNothing`/`onConflictDoUpdate` target).
  - **`etl/lib/pseMarketWatch/`** — `PseMarketWatchForeignFlowSource` (used only by `fetch-foreign-flow.ts`) extracts real index-level Foreign Buying/Selling from PSE's free weekly "Market Watch" PDF (linked from `pse.com.ph/market-report/`; the link itself is discovered from that page's HTML, since the PDF filename encodes a week number that isn't otherwise predictable). The PDF's raw text stream interleaves the table with footer/disclaimer content rather than reading row-by-row, so `parseMarketWatchPdf.ts` does position-based extraction with `pdfjs-dist` (text items grouped into rows by y-coordinate, ordered into columns by x-coordinate) instead of a naive text regex.
  - **`etl/lib/pseQuotationReport/`** — `PseQuotationReportBlockSaleSource` (used only by `fetch-block-sales.ts`) extracts real block sale trades from PSE's free **daily** "Daily Quotation Report" ("End of Day Quotes" on `pse.com.ph/market-report/`, filename `<Month> <D>, <YYYY>-EOD.pdf`, link discovered from that page's HTML the same way as the weekly Market Watch PDF). Same position-based `pdfjs-dist` extraction approach as `parseMarketWatchPdf.ts` — the BLOCK SALES table sits between two near-identical "SECURITY/PRICE/VOLUME/VALUE" header rows (PHP-denominated, then a USD-denominated variant), not delimited by anything a text regex could key off. Only the PHP table is parsed (`BlockSale` has no currency field). The report has no per-row date — one PDF is one trading day, so `tradeDate` comes from the filename.
    - **2026-07: this same report also has a real per-stock "Net Foreign Buying/(Selling), PHP" column** — `parseQuotationReportForeignFlow` (`parseQuotationReportPdf.ts`), also called from `fetch-block-sales.ts` on the *same* PDF fetch (not a separate request — `pse.com.ph`'s WAF is flaky enough without doubling requests for data in the same file). Found by re-investigating the long-standing "per-stock foreign flow has no free source" conclusion (see above) — every MAIN BOARD page of the report lists one row per stock (Issue Name, Symbol, Bid, Ask, Open, High, Low, Close, Volume, Value, Net Foreign Buying/(Selling)), sector-grouped under spaced-letter headers ("F I N A N C I A L S") and "SECTOR TOTAL" footers. Rather than parse the column header (it's split across 3 different y-coordinates — "Net Foreign" / "Buying / (Selling)," / "PHP"), a stock row is found structurally: a ticker-shaped item in the ticker column's x-range (100–160, verified live), and the value is always the rightmost item on that row. `PseQuotationReportBlockSaleSource.getLatest()` calls `getDocument()` **twice on separate `pdfBytes.slice()` copies**, not once shared — pdf.js's `getDocument({ data })` transfers/detaches the underlying buffer on use, confirmed live (a second call on the same `Uint8Array` throws "Unable to deserialize cloned data" even called sequentially, not just under `Promise.all`). Only the top/bottom 10 by net value are stored in `stock_foreign_flow` (`getLatestStockForeignFlow` in `packages/db`), keyed by the report's actual trade date now (real daily granularity, not the old mock's weekly-Friday convention) — **written via delete-then-insert per `periodEnd`, not upsert**: this table is a top-N leaderboard, so a ticker that drops out of today's top 10 needs its old row *removed*, and `onConflictDoUpdate` only touches rows whose key still matches something in the new insert, leaving stale entries from a previous write (e.g. the old mock's hardcoded tickers) sitting next to real ones whenever `periodEnd` happens to collide — confirmed live, since the mock's weekly-Friday and the real daily-Friday landed on the exact same date at least once.
  - **Both PDF sources share `etl/lib/fetchWithRetry.ts`.** `pse.com.ph` itself (not just `edge.pse.com.ph`) is genuinely flaky — confirmed live by hitting `pse.com.ph/market-report/` 5 times in immediate succession with identical headers: 2 succeeded (200), 3 failed (500). Also confirmed it's a real WAF quirk, not caller error: a request with no `Accept` header 500s regardless of User-Agent (a full browser UA alone still 500s), but adding `Accept: text/html,application/pdf,*/*` fixes it *some* of the time — so both the header and the retry are load-bearing. Any new source hitting `pse.com.ph/market-report/` (as opposed to `edge.pse.com.ph`, which hasn't shown this) should go through `fetchWithRetry`, not a bare `fetch()`.
  - **These live in `etl/lib/`, deliberately *not* in `packages/sources/foreign-flow` or `packages/sources/block-sales`.** `pdfjs-dist` has a real module-init side effect (a `DOMMatrix` polyfill check) that isn't tree-shakeable once re-exported through a package's barrel `index.ts` — putting it in a shared package broke static generation for *every* `apps/web` route (silent `DOMMatrix`/`Path2D` warnings during `next build`, confirmed by testing) even though `apps/web` never calls that code. Any future source needing a heavy/native-touching dependency (PDF parsing, image processing, etc.) belongs in `etl/lib/` or its own narrowly-scoped package — never in a `packages/sources/*` package whose barrel `apps/web` already imports for its types/mock. Pinned to `pdfjs-dist@4.0.379` specifically (not the latest) — newer majors require Node ≥22, while this repo's workflows and local tooling are on Node 20.
- **`packages/config`** (`@pseye/config`) — shared base `tsconfig.json` extended by every other package.
